Alternate FizzBuzz Solution

If you aren’t familiar with what ‘FizzBuzz’ refers to, see http://c2.com/cgi/wiki?FizzBuzzTest.

Q: So here is a description of the ‘FizzBuzz’ problem. Do you think you could give us a quick solution for this?

A: Sure, that looks pretty simple. I can solve it any way I like?

Q: Of course.

A: Great. Well, I’ll start by writing a unit test.

Q: … Well, that probably isn’t necessary…

A: Maybe not, but I want to be able to instantly verify the solution. Let’s see. Well, to test the output, I’ll need a reference string. [Begins writing] Something to compare the output to show that it’s correct… So, like, “1 2 Fizz 4 Buzz Fizz 7 8 Fizz Buzz 11 Fizz 13 14 FizzBuzz…” [Continues writing]

Q: Well… Okay…

A: Does this look right? [Displays finished string of numbers 1..100 with Fizz & Buzz appropriately inserted] As in, if our output matches this, we’ll have a successful program?

Q: Sure, that looks correct, but…

A: Great. Now let’s write our function. Here’s our method; we’d run the test here and watch it fail. Now, just to make it green, I’ll just have the method output our reference string.

Q: …

A: Say, this actually fulfills the requirements right now. I guess I’m done.

Q: But you actually haven’t written anything to programmatically produce the FizzBuzz string!

A: True, but since we already have a string which we’ve agreed is the correct result, if we just output that string, we’ll satisfy the requirements for the problem. So, I’d say we’re done.

Q: But what if we wanted to see the results from 1 to 10,000?

A: Well, gee. That wasn’t in the requirements.

The Real Reason We Write Bad Software →

… even the best of these APIs are hard to use because the documentation, supposedly written in English, is terrible. Most engineers can’t write a single coherent sentence, never mind string together a paragraph.

I think I’d agree with this, but with reservations. For starters, I think many of the developers I know are better than average writers. Maybe I’m in an anomolous community, but that’s my impression. Second, just because someone can write well, doesn’t mean they will write any documentation, let alone good documentation. It’s food for thought, though.
